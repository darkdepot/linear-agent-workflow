## A — Issue-only lifecycle + start-time audit + promotion

**New entities.** `linear-issue-intake` = NEW atomic front-door skill for genuinely one-PR projectless work (collapses idea→discovery→handoff→issue into one gate; NOT a redirect adapter — must clear validator's real-skill floor). Route-record lives in a comment on the Issue (`artifact_profile:issue-only, container:issue, required_artifacts:["issue"], oracle_kind:issue-verification`). Front-door selector = the resolver (both linear-idea and linear-issue-intake call resolver first; `resolver.artifact_profile` decides owner); profile auto-inferred at intake (explicit one-PR / no product surface / no new actors ⇒ issue-only; else project-first; ambiguity ⇒ project-first fail-closed).

**Circularity fix (Sol R8).** Approval anchors on `scope_fingerprint = H(scope ∥ acceptance-with-stable-IDs ∥ non-goals)`, NOT a draft object. Transaction (ordered, each fail-closed): 1 preview (nothing written, resolver=needs-record) → 2 start-time audit over preview text → 3 approve the fingerprint → 4 create Issue (body reproduces scope/acceptance/non-goals verbatim; recompute fingerprint) → 5 guard (recomputed≠approved ⇒ delete/park, BLOCKED, re-preview) → 6 write route-record `route_revision=1` with `approval_refs:[{type:impl-start, ref:issueId, fingerprint, route_revision}]` + audits → 7 start (implement verifies recorded approval vs live fingerprint). Chain durable+acyclic: fingerprint→Issue(re-verified)→route-record(refs Issue id)→start(verifies).

**Oracle.** `oracle_kind=issue-verification`, `oracle_id=issueId`, `oracle_revision`=hash of normalized `# Критерии приемки`+`# Как проверить`; each AC has stable ID AC1..ACn; each verify step names its AC(s). Deploy Live-QA reads `resolver.behavioral_oracle`: issue-verification ⇒ walk AC1..ACn at pinned oracle_revision; asserts `oracle_drift==none` (else needs-human). Design-acceptance skipped for issue-only when no prototype.

**Stage break table (branch on resolver OUTPUTS, never raw profile string):** intake (linear-idea:29,51 Project-mandatory ⇒ resolver routes issue-only to linear-issue-intake; linear-idea gains a guard line) · issue-drafting (linear-issue:12,27,38 requires Project/PRD/Spec+redirects ⇒ profile-gated create mode when required_artifacts==["issue"]: no redirect, no chips) · approval (artifact-rules:45 ⇒ approval object = scope_fingerprint) · route-record storage (net-new: first authoritative comment at creation) · dispatch (orchestrator-dispatch:47-54 requires Project/PRD/Spec ⇒ snapshot profile-shaped; PRD/Spec render n/a) · implement (linear-implement:57 moves Project ⇒ branch on lifecycle_state_source; issue-only moves the ISSUE; check delivery issue-only mode) · preflight (linear-preflight:49 compares vs Project/PRD/Spec ⇒ compare vs behavioral_oracle+scope_fingerprint; may RAISE new route_revision) · ship (linear-ship:29 routes parentless Issue to handoff ⇒ gate becomes "no valid route-record+approval ⇒ intake owner") · deploy (linear-deploy:28,63 fetch+walk PRD ⇒ shaped by required_artifacts; walk issue-verification ACs) · resume (orchestration:452 scans projects-in-flight ⇒ add issue-only query).

**Two audits (Sol R7 — preflight is NOT independent: one worker runs implement→preflight→ship same session).** START-TIME AUDIT by a fresh classifier BEFORE express approval, over preview text: topology_check (can a zero-context executor implement w/o inventing actors/behavior/states/acceptance/non-goals — with stable AC IDs) + documentary_closure_check (each AC implementable-from-text AND observable-live; CANNOT be recomputed from a diff → must be pre-start) + initial assurance_check (run reducer). Run by the orchestrator session / a dedicated fresh-context classifier it spawns at the approval stage; the worker doesn't exist yet ⇒ collapse impossible by construction, PROVIDED documentary-closure is never deferred to preflight. Explicitly NOT a cross-vendor Second Voice — structural/deterministic classification, cites code-review-single-vendor memory; must be a different context than the drafter. PREFLIGHT AUDIT on the actual diff: recomputes assurance_check ONLY (never topology/documentary — frozen from start-time); may RAISE + new route_revision; rejects unrecorded hard-trigger mismatch. Detector ONE-WAY; path patterns CANNOT infer pii/cross-tenant/public-contract/shared-db-blast-radius → absence=absence-not-proof; those classes gated by documentary_closure_check + ioc floor, not the path detector.

**Promotion (Issue→Project).** Preserve branch+PR; preserve old certs as HISTORY only; NEVER preserve validity across profile promotion. Cert MUST bind {head_sha, base_sha, route_revision, scope_fingerprint, artifact_profile, oracle_kind, oracle_revision, assurance_ruleset_version, route_schema_version}; honored iff every bound field == live resolver output (fail-closed). Timing: before any preflight cert ⇒ promote, rerun delivery readiness under project-first, oracle flips prd-acceptance ⇒ impl-start approval invalidated, produce PRD/Spec via handoff. AFTER a `ready` cert — SIMPLEST SAFE CUTOFF: do NOT promote in-flight slice; freeze issue-only, ship+deploy it (frozen cert stays valid), create a separate follow-up Project for expanded scope; only a safety defect revokes. Start-approval survives freeze iff live fingerprint == approved fingerprint. **Open-Q1 (precedence): Issue inherits container record; may override only with explicit override_reason naming the superseded container route_revision; bare override = fault (fail-closed inherit+flag).** **Open-Q4 (observed-live write-back): yes-for-learning (new route_revision flagged post-close-learning, may lower residual for learning/labels), no-for-re-gating a closed unit.**

**Fixtures (10, subprocess pattern):** creation (issue-only vs project-first required_artifacts/lifecycle_state_source/oracle_kind) · delivery (moves Issue not Project) · preflight (unrecorded HT ⇒ assurance_check fail + route raised) · ship (parentless valid record ⇒ not routed to handoff; no record ⇒ fail) · deploy (oracle_kind=issue-verification walk-list=ACs; oracle_drift⇒fail) · promotion-before-preflight (approval invalidated) · promotion-after-ready (freeze not re-gate; follow-up Project) · stale-approval (fingerprint mismatch⇒fail) · stale-certificate (any bound field≠live⇒fail) · resume-discovery (issue-only query returns it; projects-in-flight scan empty; unverified label⇒fail).

## B — Containment-first operational safety

**Consumes spine (never re-derives risk):** reads ioc, HT, safety_modules[] (Step D), deploy_approval (Step E, INV-2), rollout/monitoring (Step F), residual proof_state. Owns runtime deploy behavior.

**§1 Fork then sequence (Phase 0 Classify):** non-destructive+reversible (persistent state & external side-effects unaffected) ⇒ fix-forward (existing default). destructive-data/irreversible/payments/auth/pii/cross-tenant/shared-db OR unknown-blast-radius ⇒ containment-first (unknown fails closed to containment). **Sequence A–E (fixed order):** A stop/disable/isolate (halt the bleed — fire the reviewed abort action) · B establish+PROVE-restorable last safe checkpoint (none ⇒ owner-accept data-loss, escalate) · C roll back code/traffic ONLY when compatibility proven (code-only ⇒ safe; schema shipped ⇒ forbidden unless expand-only+target-code-proven OR schema first reverted, else skip to D; irreversible emitted ⇒ can't undo, go D) · D restore (to checkpoint) OR forward-repair (corrective migration — itself re-enters gates as a new risky unit) · E verify content invariants ALWAYS last (row counts, referential integrity, reconciliation, no-double-charge, tenant-isolation — the state-level analogue of live-QA). Amend linear-deploy:65 + lifecycle:188 with the fork (fix-forward retained verbatim as else-branch). **Live-QA amendment:** data-integrity units require invariant verification in addition to functional smoke (closes "UI smoke green while 2% data corrupt").

**§2 Progressive bundle (evidence ladder IS spine proof_state):** Issue/Tech-Spec (planned invariants, recovery branches restore-vs-forward-repair + whether rollback is even possible, abort thresholds) [planned] → preflight (staging dry-run forward+rollback proven, backup/restore exercised, reconciliation balanced, isolation green) [verified-predeploy + evidence_ref] → ship (bind evidence to final head SHA; re-confirm on new head) → deploy (verify target==certified SHA, checkpoint restorable NOW, approval per Step E, monitoring authority; live-QA ⇒ observed-live). Monotonic; planned never lowers anything (INV-1); observed-live lowers monitoring only.

**§3 Safety certificate** (marker `linear-safety certificate`, comment, Russian lead, most-recent-wins): binds route_revision/schema/ruleset versions, ioc, HT, head_sha, merged_sha (must equal at deploy), deploy_target, schema_version from→to, migration_version, residual+per-module proof_state+evidence_timestamp, safe_checkpoint{kind,ref,restore_verified}, recovery_plan{default_branch,abort_action,abort_thresholds,invariants}, recovery_authority{holders:[orchestrator,owner],scope:[phase-A,phase-C-when-compat-proven],expires_with_head_sha}, deploy_approval, rollout/monitoring, live_verification. Fail-closed arming: destructive/irreversible can't reach `armed` with checkpoint.kind=none / restore_verified=no / any mandated module still planned.

**§4 Emergency = sequencing profile, NOT container/requirements.** Urgency never lowers route/review/safety_modules/deploy-approval-requirement. deploy-approval GRANTS standing containment authority (orchestrator+owner — our only operators) for Phase A + compat-proven Phase C (stopping/damage-reducing only); forward-repair or data-loss-accept still need a fresh decision. Authority bound to head_sha, stale on new head. Every containment action = mandatory ledger line. Fast path buys the right to STOP immediately, never to skip B/D/E.

**§5 Issue-only migration bundle:** must be Issue-attached, durable, recoverable by a fresh linear-deploy, bound to same route_revision+head_sha; storage-shape deferred to D (resolved there as Issue-modules).

**Fixtures (8):** classifier fork (incl unknown⇒containment) · sequence ordering (code-rollback on schema-mutation before compat-proof rejected) · compat gate · bundle monotonicity + INV-1/INV-2 RED · emergency doesn't lower container · recovery-authority staleness · cert fail-closed arming · cert↔route-record binding+recovery.

**Deferred:** per-target abort-action vocabulary (→ deploy-config); ruleset-versioning of safety-module semantics (→ spine #3 / governance).

## C — Initiative tier

**Definition.** Initiative = multiple distinct deliverables bound by a shared OBJECTIVE (NOT a shared foundation). is_outcome ⟺ owner can accept+release+rollback independently ∧ value survives cancel(all siblings) ∧ not a mere internal checkpoint (need NOT be user-facing — security/maintenance/platform qualify). REQUIRES ≥2 outcome children; 1-outcome = smell ⇒ topology_check fail.

**Child roles.** outcome (counts) vs enabler (shared prerequisite, value realized only through outcomes it enables, fails survives-cancellation ⇒ can't count). role authoritative in initiative route-record `child_topology[].role`; hint-only label `initiative-role/*`. **Operational independence:** each outcome rolls back without breaking a sibling (rollback target stays in dependents' provider_version_range); enabler evolves backward-compatibly across live consumer versions (supports union of ranges); every depends-on edge carries a compat-checkpoint {provider_ref, contract_ref, provider_version_range, checkpoint_state:pending|verified|broken, evidence_ref when verified}.

**Foundation vertical-first (repo's "two adapters make the seam real", execution-quality:121).** Default: ship/spike one thin integration → observe a 2nd consumer's variance → extract seam once TWO OBSERVED consumer contracts. Enabler-first only with all 5 gates in `enabler_justification`: two observed consumer contracts + explicit enabler role + backward-compat dependency + pre-impl duplication evidence + post-impl deletion test.

**Aggregate cutover risk (NOT max(child.route) — that floods unrelated children).** A `cutover_group` = named subset sharing a coordinated go-live (0/1/many per initiative; child in no group ships fully independently). Initiative record carries a top-level vector (the brief-authoring unit's own low assurance) AND per-group `emergent_vector` (rc=choreography difficulty not a rollup; ioc=the combined-flip consequence belonging to no single child, may exceed any child or be none; rr; HT firing only at aggregate: shared-db-blast-radius/combined-public-contract/cross-tenant). Same spine reducer runs on emergent_vector ⇒ group_assurance_requirements; gate attaches to the GROUP CUTOVER EVENT (a linear-deploy coordinated release), never to children's records. Preserved floors apply at group level (migration⇒R3, irreversible⇒R4). Two escalation channels: child's own preflight RAISES the child; group emergence gates the group. Group deploy_approval keys on ioc+HT+policy never rr (INV-2). Deploy consumes via resolver (linear_ref=initiative, stage=deploy) ⇒ live-QA extends to combined surface + DAG-edge compat-checkpoints.

**DAG enforcement WITHOUT orchestrator stage work.** Today orchestrator honors only Issue deps (linear-orchestrate:116). Mechanism: (1) resolver surfaces child_topology/cutover_groups on container resolution (additive, co-versioned); (2) orchestrator SEQUENCES on edge checkpoint-state reusing the existing "queue dependents until blockers done" — keyed on Project-edge+checkpoint-state — this is control-plane sequencing, not stage work; (3) WORKERS produce checkpoint evidence (consumer's preflight/ship verifies against provider's pinned contract version) as ordinary stage work; single-writer orchestrator applies the pending→verified mutation (verify-after-write). Control-plane boundary intact.

**Closure.** initiative_closed ⟺ objective_metric_satisfied ∧ owner_accepted (independent of children-done: all-done+metric-unmet ⇒ open; metric-met+child-cancelled ⇒ closeable). Stop/cancel: met-early (cancel surplus — outcomes independently cancellable) or proven-unreachable (shipped outcomes survive, not rolled back). Acceptance authority = owner; orchestrator routes the closure/cancel brief, never self-closes.

**Brief = native Linear Initiative description**, WHY+orchestration ONLY, NO architecture (ADR/Tech-Spec by link), no machine block. Sections (RU at impl): Цель · Критерий завершения (metric+stop/cancel+authority) · Результаты (outcome/enabler tags + DoD) · Зависимости (human DAG) · Кросс-проектные ограничения · Условия остановки · Решения и спецификации (links).

**Ownership.** NEW atomic skill `linear-initiative` (mirrors linear-project one tier up): owns the Initiative description + container route-record (child_topology, cutover_groups, objective, roles) + labels + topology_check. NEW modes `linear-review initiative` (structure review, report-only) + `linear-check initiative` (3 container audits, readiness-only). Closure authority=user; orchestrator routes+delegates the mutation to linear-initiative. Justified vs AGENTS.md (adds one-artifact skill, doesn't collapse delivery skills, keeps control-plane boundary). Lifecycle: linear-initiative at portfolio framing (one tier above linear-idea) → children via normal idea→handoff linked with roles → orchestrator coordinates honoring DAG → linear-deploy runs cutover-group releases → owner accepts vs metric → linear-initiative records closure.

**Schema additions (additive route_schema_version bump, fail-closed):** objective{statement, closure_metric{definition,target,measurement_source,window}, acceptance_authority}; child_topology[]{project_ref, role, depends_on[]{provider_ref,contract_ref,provider_version_range,checkpoint_state,evidence_ref}, enabler_justification?}; cutover_groups[]{group_id, members[], emergent_vector, group_assurance_requirements}. ## D — Research/spike lifecycle + projectless-document placement

**Orthogonal axis.** `execution_profile` (NEW route-record field) = delivery | research — terminal-output shape, ORTHOGONAL to artifact_profile (topology). 2×2 valid. Missing/unknown ⇒ delivery (fail-closed; research must be positively declared). Stages never branch on raw string (resolver expands).

**Research lifecycle.** One-terminal-output law: delivery Issue → exactly 1 PR; research Issue → exactly 1 evidence/decision closeout; never both. Research may write throwaway probe/prototype code as EVIDENCE (never advanced to preflight/ship/deploy); if code should ship it spawns a separate delivery Issue. This FIXES "gates constant on every container" — gates are resolver-derived per execution_profile: delivery=code-closure (autoreview+live-QA), research=evidence-closure (certificate completeness); both rigorous, different-shaped. **Entry-honesty test:** admit research ONLY for a knowable fact we lack (backend enum? feasibility? reversible-in-lock-window?), NOT a preference/tradeoff we must choose (that's discovery/handoff; linear-implement exits needs-human/needs-decision if the "unknown" is a decision). Stop: threshold met OR budget exhausted (never open-ended). **Inconclusive = DONE** ("no viable path" saved a doomed delivery). Terminal statuses (disjoint from implement's, can't fall into implemented-needs-preflight): answered-viable (Done), answered-inconclusive (Done), blocked (no), needs-human (no). **Certificate** (marker `linear-research certificate`, comment, RU lead): Research status, decision question, evidence threshold+met, method, evidence (concrete re-inspectable, never "I concluded"), confidence+basis, **disconfirming evidence MANDATORY (may not be none — adversarial discipline)**, recommendation (viable|viable-with-constraints|no-viable-path|blocked-external|needs-decision), consumer (exact downstream slot), invalidation keys, not-proven, next. **Consumer** (exactly one): Tech-Spec section (esp Реальные ответы бэкенда) / PRD Допущения-Открытые вопросы / ADR / handoff estimate — updated as closeout's final act. **Terminal path NEVER code preflight/ship/deploy** — invoking them on a research unit fails-closed with the redirect. **Closure criterion (discharges spine open-Q5, oracle_kind=none):** DONE iff pre-registered decision_question answered to evidence_threshold (or valid inconclusive), certified by a complete linear-research certificate, verified by NEW `linear-check research-closeout` mode. behavioral_oracle.kind=none; NEW sibling `closure_oracle={decision_question_ref, evidence_threshold_ref}`.
